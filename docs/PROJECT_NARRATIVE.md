# 项目思路与讲述方式

本文档用于帮助梳理“中级阶段拉姆桌面宠物”这个项目的开发逻辑、工程判断和对外讲述方式。

## 项目一句话

这是一个用 Vue 3 + Electron 做的本地 macOS 桌面宠物 App，把中级阶段拉姆从静态素材整理成一个可拖拽、可互动、会自动活动并能长期悬浮陪伴的桌面应用。

## 你的项目思路是否清晰

整体思路是清晰的，而且符合一个小型产品从原型到可用版本的正常路径：

1. **先定义产品目标**
  - 不是做一个复杂养成游戏，而是先做一个轻量、低打扰、可长期打开的桌面陪伴工具。
  - 这让你暂缓了数值系统，避免过早把项目变复杂。
2. **先打通桌面应用能力**
  - Electron 负责透明窗口、置顶、托盘、右键菜单、窗口位置持久化。
  - Vue 负责桌宠本体、状态切换和动画表现。
  - 这个分层合理：系统能力放主进程，交互表现放 renderer。
3. **再整理素材和状态**
  - 先把 `src/assets/ram/` 中的单张 PNG 作为 App 运行素材。
  - Codex spritesheet、hatch 产物和历史素材放进 `codex-related-export/`，避免和正式 App 运行链路混在一起。
  - 后续又修正了 `dirty` 和 `walk-2` 的素材问题，说明你在按实际体感迭代，而不是只看功能清单。
4. **最后收敛交互规则**
  - 你没有继续盲目加情绪，而是开始问“点击会出现什么、自动行为频率如何、状态之间如何覆盖”。
  - 这一步很关键，因为桌面宠物的质量主要不在情绪数量，而在它是否自然、不烦人、可预测。

## 是否符合开发标准

当前已经符合小型本地 App 的基本工程标准：

- 有清晰目录分工：`electron/`、`src/`、`docs/`、`scripts/`、`src/assets/ram/`。
- 有运行时状态集中定义：`src/lib/pet-state.ts`。
- 有交互逻辑文档：`docs/INTERACTION_LOGIC.md`。
- 有情绪状态清单：`docs/EMOTION_STATES.md`。
- 有 Roadmap、Testing、Implementation Notes 和 Changelog。
- 没有把未稳定的新情绪直接接入主状态机，降低了回归风险。

还需要补强的地方：

- 本地构建环境还不稳定，`npm`、`rolldown`、`sharp` 需要修复后才能形成可靠验证链路。
- 当前没有自动化测试，主要依赖手动体验验收。
- 状态机参数仍是体感值，需要经过实际使用后定稿。
- 尚未做 `.app` / `.dmg` 打包和开机启动，这些属于后续发布能力。

## 推荐讲述结构

如果你要向别人讲这个项目，可以按下面顺序讲。

### 1. 背景

我想做一个中级阶段拉姆的桌面宠物，不是网页预览，而是一个可以长期挂在 macOS 桌面的轻量陪伴 App。核心目标是“可爱、稳定、不打扰”。

### 2. 技术选型

我用 Electron 做桌面能力，包括透明无边框窗口、置顶、托盘菜单、窗口位置持久化；用 Vue 3 做宠物本体的状态渲染、点击/拖拽交互和动画。

### 3. 架构分工

主进程负责系统级能力：窗口、托盘、菜单、配置读写。Renderer 负责表现层：当前 mood、状态切换、动画、素材加载。状态和素材路径集中在 `pet-state.ts`，这样新增或调整状态时不会散落在各处。

### 4. 迭代过程

第一步先做基础桌面宠物：透明窗口、置顶、拖拽、点击反馈。第二步加入托盘和右键菜单，让它像一个真正的桌面 App。第三步整理素材和状态，修复脏污图抠图、走路帧体积跳变。现在进入第四步：收敛交互规则，让它的行为节奏自然。

### 5. 关键产品判断

我暂时没有接入复杂的饥饿、清洁、心情、健康数值系统，因为这会把项目从轻量陪伴推向养成游戏。当前阶段先保证桌面体验好，再决定是否恢复数值系统。

### 6. 当前状态

现在已经有 17 个接入状态，包括待机、开心、走路、睡觉、抱起、学习、工作、玩耍等。点击会触发积极情绪，拖拽会进入抱起状态，空闲时会自动走动、随机学习/工作/玩耍，长时间无交互会睡觉。

### 7. 下一步

下一步不是继续加素材，而是做交互体感验收：自动走路是否太频繁，随机情绪是否自然，睡眠时间是否合适，点击反馈是否稳定。等行为规则稳定后，再考虑打包成 `.app` / `.dmg` 和加入开机启动。

## 30 秒版本

我做的是一个 Vue 3 + Electron 的本地桌面宠物 App。Electron 负责透明悬浮窗、置顶、托盘菜单和配置持久化，Vue 负责拉姆的状态机、点击拖拽和动画表现。项目不是一开始就做复杂养成，而是先把轻量桌面陪伴体验做好：能拖动、能互动、会自动走动、会睡觉，同时保持不打扰。现在素材和基础能力已经基本完成，下一步是调状态机参数，让它的行为节奏更自然。

## 合作方介绍版本

我现在做的是一个中级阶段拉姆的 macOS 桌面宠物。它已经不是静态图片 demo，而是一个能长期悬浮在桌面上的本地 App：可以拖拽、点击互动、自动走路、睡觉，也已经接入了 17 个角色状态。

当前阶段的重点不是继续堆状态数量，而是把桌面陪伴的体感调自然：什么时候走动、多久睡觉、点击反馈是否稳定、右键菜单是否符合用户直觉。这个阶段做扎实后，后面再接入更完整的养成系统。

下一步我想做的是数值驱动的状态变化，比如饥饿、清洁、心情、健康。这样拉姆的状态不是随机出现，而是有“生活状态”作为原因。比如饿了会出现 `hungry`，喂食后进入 `eating`；脏了会出现 `dirty`，清理后进入 `cleaning`；心情低落时会出现 `sad`，用户互动后恢复到开心或亲密状态。

再往后可以做用户状态提醒和性格变化。提醒不一定用弹窗，而是通过拉姆自己的状态和动作表达；性格则可以让不同用户拥有不同风格的拉姆，比如活泼型、安静型、黏人型、学习陪伴型。

从合作角度看，这个项目不只是一个单独桌宠，而是一套“角色素材 + 行为状态机 + 桌面陪伴体验”的框架。后续可以接不同角色、不同 IP、节日状态包、皮肤包、学习/工作陪伴模式或品牌联名桌宠。

## 面试/展示版本

这个项目我比较重视的是“从素材原型到桌面产品”的完整链路。早期素材来源比较杂，有 Codex spritesheet、生成图和单张 PNG。我先把运行时素材收敛到 `src/assets/ram/`，把历史导出放到归档目录，降低运行链路复杂度。实现上，Electron 主进程只处理系统能力，Vue renderer 只处理宠物状态和展示，状态枚举和素材路径集中维护。后续我又把交互逻辑文档化，明确每种用户行为和自动行为会触发哪些 mood，以及限时状态如何阻止自动行为。这样后面无论是调参数、加新情绪还是恢复数值系统，都有比较清楚的边界。

## 当前最应该强调的取舍

- 先做桌面体验，再做养成系统。
- 先收敛状态机，再扩展情绪数量。
- 先保证素材体量一致，再谈丰富表现。
- 先文档化当前行为，再做下一轮交互调整。
