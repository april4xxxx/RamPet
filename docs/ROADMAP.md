# 中级阶段拉姆桌面宠物 Roadmap

## 当前目标

把拉姆从开发预览项目升级成可长期本地使用的 macOS 桌面悬浮宠物 App。

当前产品定位是“轻量陪伴、不打扰、可长期挂在桌面”。现阶段先把桌面体验、交互节奏和状态表现做稳定，再考虑更完整的数值养成、性格系统和发布能力。

## 当前阶段

状态：交互规则验收与产品口径定稿。

素材修复、核心桌面能力和基础状态机已经基本完成。当前重点不是继续扩展情绪数量，而是确认现有 `PetMood`、自动行为频率、状态覆盖规则和菜单交互是否符合“长期陪伴”的体感。

## 已完成

### 桌面 App 基础能力

- 透明无边框 Electron 悬浮窗。
- 保持置顶，并支持从菜单切换置顶状态。
- 可拖拽移动，窗口位置会持久化。
- 防止窗口拖出当前屏幕工作区。
- 托盘/菜单栏图标。
- 托盘左键和右键都会弹出同一菜单，避免“点击到底是显示还是切换置顶”的歧义。
- 显示/隐藏拉姆、重置位置、退出。
- 应用启动加载完成后主动显示拉姆，不再被上次隐藏状态阻止。

### 当前交互与自动行为

- 单击宠物：从 `affection` / `happy` / `play` / `excited` 洗牌池依次触发，一轮内尽量不重复。
- 悬停宠物：仅刷新交互时间；如果正在睡觉，则唤醒回 `idle`，不额外触发安抚情绪。
- 拖拽宠物：移动超过 `4px` 后进入 `carried`，松手后回 `idle`。
- 自动走路：每 `9000ms` 尝试一次，单次走路 `1920ms`；桌面模式下每 `60ms` 位移 `6px`。
- 走路方向：70% 维持上次方向，30% 反向，减少左右横跳。
- 撞到屏幕边缘：本轮 `walk` 立即结束并回 `idle`，下一轮反向。
- 环境随机：每 `10000ms` 从 `idle` / `study` / `work` / `play` / `excited` 中抽样。
- 环境状态持续时间已分档：`study=8000ms`、`work=9000ms`、`play=3000ms`、`excited=2200ms`。
- 闲置睡眠：无交互超过 `300000ms`（5 分钟）后进入 `sleep`。

### 菜单动作

菜单中已支持：

- 摸摸：触发 `affection`。
- 玩耍：触发 `play`。
- 学习：触发 `study`。
- 工作：触发 `work`。
- 睡觉：触发限时 `sleep`。
- 走路（测试）：触发真实窗口位移的 `walk`，用于快速验证走路链路。

## 当前未完成或需要继续确认

### 交互体感验收

这些不是代码缺失，而是需要真实使用后继续确认：

- 自动走路每 `9000ms` 尝试一次是否仍然过于频繁。
- 走路单次位移约 190px 是否自然，是否需要按屏幕大小或用户偏好调整。
- 环境随机每 `10000ms` 尝试一次是否足够自然。
- `study=8000ms`、`work=9000ms` 是否太长，是否会让拉姆看起来“卡住”。
- 闲置 5 分钟睡觉是否符合桌面陪伴节奏。
- 悬停唤醒是否需要增加短暂过渡表现，例如 `wake`、伸展、朝向鼠标轻摆。
- 右键/托盘菜单的动作是否覆盖了最常用的主动交互。

### 状态与素材缺口
- `hungry`、`dirty`、`sick`、`sad`、`eating`、`cleaning`、`medicine` 已在 `PetMood` 中存在，但当前没有自动触发路径。
- `eating` 目前没有独立素材，代码临时映射到 `idle`。
- `jumping`、`spotted`、`waving` 有素材文件，但未纳入 `PetMood` 和状态机。
- `calm`、`thirsty`、`shocked` 等仍是候选方向，不应在当前阶段承诺为已接入状态。

- 当前 shell 中 `npm` 不可用，`npm run build` 无法执行。

- 修复环境后再跑 `npm run desktop:prod`，确认 `file://` 打包路径、素材加载和 Electron 桌面行为正常。

## 后续迭代

### 偏好设置
- 性格或陪伴风格选择。
  - 自动走路频率。
  - 睡眠等待时间。
  - 是否保持置顶。
  - 是否开机启动。
  - 是否启用提醒。

### 数值驱动的陪伴系统
把已预留的 `PetCareStats` 接入真实状态变化，让拉姆不只是随机切换表情，而是由饥饿、清洁、心情、健康等数值驱动：
- 饥饿降低时触发 `hungry`，用户喂食后进入 `eating`。
- 清洁降低时触发 `dirty`，用户清理后进入 `cleaning`。
- 健康降低时触发 `sick`，用户照顾后进入 `medicine` 或恢复状态。
- 心情降低时触发 `sad`，用户互动后恢复到 `happy` / `affection` / `play`。

### 用户状态提醒
将提醒做成桌面宠物的自然行为，而不是强打扰弹窗：
- 拉姆通过状态变化提醒用户该互动、休息或处理某个需求。
- 低频、可关闭、可调节，保持“轻量陪伴、不打扰”的产品原则。
- 后续可以扩展到学习、工作、休息等场景提醒。
- 下一步计划：接入 API，让提醒内容和提醒时机从固定文案升级为基于当前上下文的判断。
- API 加入 memory，提醒事项。

### 拉姆性格变化
在状态机之上增加性格层，让同一个拉姆在不同用户那里有不同陪伴风格：
- 活泼型：更容易走动、玩耍、主动出现积极状态。
- 安静型：更少打扰，更偏向陪伴、睡觉、学习、工作状态。
- 黏人型：更重视点击、悬停、长时间无互动后的反馈。
- 学习/工作陪伴型：更多触发 `study` / `work`，适合专注场景。

### 新状态与素材扩展
新增状态必须绑定对应图片或动画素材，避免只扩展状态名但没有稳定表现：
- 可优先考虑 `wake`、`waving`、`jumping`、`spotted`、`curious`、`shy`、`bump`、`thirsty` 等高辨识度状态。
- 每个新状态需要定义触发条件、持续时间、退出规则和素材规格。
- 状态扩展应服务于交互体验或数值系统，不单纯追求数量。

### 发布与合作准备
- macOS `.app` / `.dmg` 打包。
- 开机启动。
- 公开发布前的版权与素材合规处理。
